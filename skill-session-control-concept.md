# Концепция: Управление скиллами в рантайме

## Проблема

Модель управляется текстом — system prompt определяет поведение. Для разных задач нужны разные сценарии (промпты):

- "исследуй сеть" → один детальный сценарий (ping sweep → nmap → анализ)
- "аудит хоста" → другой сценарий
- "разработка" → третий

Запихнуть всё в один промпт = раздутый контекст, который тащится всегда. Критично для локальных LLM с маленьким контекстным окном.

## Текущее решение: Skills

Скилл = промпт как ленивый инструмент:
- **description** (короткая) — всегда в контексте, триггер для выбора
- **SKILL.md** (полная) — загружается только при вызове

Экономия: полный сценарий не в контексте до востребования.

**Проблема текущей реализации:** все установленные скиллы видны модели всегда (все descriptions в контексте). При большом числе скиллов это само по себе занимает токены.

## Предложение: Явное включение/выключение скиллов

**Идея:** пользователь в любой момент контролирует какие скиллы активны. Неактивные — полностью исключаются из контекста (ни description, ничего).

**Реализация через профили:**

```
профиль "сетевой аудит"  → {network-scan, host-audit, vuln-check}
профиль "разработка"     → {code, git, docs}
профиль "минимум"        → {}
```

Переключил профиль → сменился набор активных скиллов в системном промпте.

## Ключевые свойства

- Модель выбирает из активных скиллов (как сейчас)
- Пользователь контролирует whitelist — явно, в рантайме
- Неактивные скиллы = нулевая стоимость токенов
- Для локальных LLM: можно работать с 2-3 скиллами под задачу и не тратить контекст

## Для агента-разработчика

Это расширение текущей архитектуры skills:
1. Добавить понятие "активен/неактивен" на уровне скилла
2. В системный промпт попадают descriptions только активных скиллов
3. UI/команда для переключения (профили или ручной выбор)
4. Опционально: профили как пресеты наборов скиллов
