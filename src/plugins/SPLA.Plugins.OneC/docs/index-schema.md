# SQLite-индекс

Индекс конфигурации хранится локально в файле `.spla/onec.sqlite`.

Мы используем SQLite, чтобы иметь:
* Детерминированное, надежное и легко перестраиваемое хранилище.
* Возможность обрабатывать огромные графы конфигураций без загрузки их целиком в оперативную память агента.
* Мощные SQL-запросы для быстрого ответа на вопросы "что использует этот объект?".
* WAL (Write-Ahead Logging) режим для быстрой вставки.

## Схема базы данных

База состоит из трёх основных таблиц.

### 1. `Objects`
Хранит узлы графа - объекты метаданных 1С.

* `Id` (INTEGER PRIMARY KEY)
* `Kind` (TEXT) — тип объекта (Document, Catalog и т.д.)
* `Name` (TEXT) — имя (РеализацияТоваров)
* `FullName` (TEXT) — Kind.Name (уникальный индекс)
* `Path` (TEXT) — относительный путь к папке объекта
* `Summary` (TEXT) — синоним объекта из метаданных

### 2. `Relations`
Хранит рёбра графа - связи между объектами.

* `FromObjectId` (INTEGER) -> `Objects(Id)`
* `ToObjectId` (INTEGER) -> `Objects(Id)`
* `Type` (TEXT) — тип связи (см. [Типы связей](relation-types.md))
* `SourcePath` (TEXT) — путь к BSL/XML файлу, где найдена связь
* `SourceLine` (INTEGER) — номер строки в BSL коде
* `Confidence` (TEXT) — `exact` (из XML) или `heuristic` (из BSL)

### 3. `Files`
Хранит хеши файлов для инкрементальной индексации.

* `Path` (TEXT) — путь к файлу (уникальный индекс)
* `Hash` (TEXT) — SHA-256 хеш содержимого
* `LastIndexedAt` (TEXT) — ISO-8601 время последней индексации

## Инкрементальное обновление

Чтобы не перечитывать тысячи файлов каждый раз:
1. `OneCIndexer` вычисляет SHA-256 каждого файла.
2. Проверяет его наличие в таблице `Files`.
3. Если хеш совпадает — файл пропускается.
4. Если хеш изменился:
   * Старые связи (`Relations`), порожденные этим файлом (`SourcePath`), удаляются.
   * Файл парсится заново, добавляются новые связи.
   * Хеш в таблице `Files` обновляется.
