# Schema-driven редактор — общий план

Цель: **абстрактный редактор структурированных файлов по переданной схеме**, agent-first.
Частный (первый) случай — редактирование JSONL-описаний таблиц БД (meta + поля) для SQL-плагина,
но сам редактор про SQL ничего не знает.

Этот док — карта. Контракты (Фаза 0) расписаны в:
- [Док А — форматы схем](schema-editor-doc-a-schemas.md) — схема данных, UI-схема, разрешение по имени.
- [Док Б — дизайн-токены](schema-editor-doc-b-tokens.md) — общий словарь стилей Avalonia ↔ Web.

---

## Зачем это вообще (исходная задача)

Документируем таблицы БД в виде «умного» JSONL: для каждой таблицы — meta-объект (правила,
связи) и список полей с `agg`/`desc`/`note`. Это даёт агенту строгие, машиночитаемые правила
(например, какие поля можно суммировать), что снижает семантические ошибки в SQL.

Наполнять эти файлы агентом удобно не всегда — в какой-то момент человеку **проще пройтись руками**
и расписать смыслы. Значит нужен человеческий редактор. Но:

- редактор — **не цель агента**. Это инструмент для человека **в обход** агентного цикла;
- схем будет много (даже для SQL — не одна), они прилетают из разных мест;
- SQL — лишь первый случай; редактор обязан быть абстрактным.

Это давно решённый класс задач — **JSON Schema → Form** (JSON Forms, rjsf, json-editor).
Берём готовый веб-рендер (вероятно **JSON Forms**) и строим вокруг него свою развязку.

## Три разные «схемы» — не путать

Слово «схема» в этом проекте значит три независимых вещи. Их смешение — главный источник путаницы.

| # | Схема | Отвечает на вопрос | Слой | Владелец |
|---|-------|--------------------|------|----------|
| 1 | **Данных** (data schema) | *что* это за данные: поля, типы, enum, ограничения | семантика | плагин/проект/агент (авторитет — поставщик) |
| 2 | **UI** (UI schema) | *как разложить* форму: порядок, группы, виджет | абстрактная подача | проект (мягкий override поверх референса) |
| 3 | **Дизайн-токены** (theme) | *как покрасить*: цвета, отступы, шрифты | визуал | общий словарь, раздельная привязка |

- **Схема данных №1 — строго стандарт JSON Schema 2020-12.** Соответствие обязательно, нестандартные
  ключи запрещены. UI (№2) — всегда отдельный документ, чтобы не «пачкать» данные нестандартным.
- enum значений (`agg ∈ {SUM,AVG,…}`) — это **схема данных №1**, а не UI. Соблазн сунуть список в UI
  («это же про виджет») неверен: тогда валидация уедет от данных. UI решает «выпадашка или радио»,
  *что в списке* — данные.
- №3 (токены) — отдельный мир от №1/№2, отдельный док (Б).

## Архитектура: стек слоёв (не альтернативы!)

Слои вкладываются друг в друга снизу вверх. Каждый нижний **не знает** про верхний и тестируется один.

```
#1 Core-редактор      — чистый компонент: Avalonia-обёртка + WebView (JSON Forms) внутри.
      ↑ встроен в        Вход: (схема данных, UI-схема?, данные) через два резолвера. Знает ТОЛЬКО их.
#3 Workspace-shell    — дерево файлов слева + панель редакторов справа.
      ↑ хостится в       Диспетчер по типу контента: .jsonl → core, .md → markdown-редактор.
#4 SPLA-host          — #3 внутри процесса SPLA; резолверы завязаны на открытый проект.
                         (тут схемы известны, ибо проект открыт)

#2 Launch-tool        — тонкий агентный вход `editor_open(file)` в любой хост. ~20 строк, не архитектура.
```

Главная причина именно такого порядка: ломается зависимость «редактор ↔ SPLA», которая иначе
превратила бы всё в монолит-IDE. Если core знает про SQL/SPLA — он больше не «абстрактный редактор
по схеме», ради чего всё затевалось.

## Сквозные принципы

1. **Agent-first, не IDE.** Окно = человеческий доступ к среде, в которой агент и так работает
   инструментами. Окно **не добавляет возможностей**, которых нет у агента. Поймал фичу окна, которой
   нет у агента, — строишь не то (IDE вместо agent-first просмотра).
2. **Два резолвера, ни один не предполагает ФС:**
   - `IJSONSchemaProvider` — схема по имени (см. Док А);
   - `IContentSource` — содержимое по ссылке (мы не гарантированно в файловой системе).
   Подробные сигнатуры — в Доке А.
3. **Рендеры независимы.** Схема несёт только семантику, **ноль визуала**. Между Avalonia и Web
   шарится *словарь токенов* (имена+значения), а не сам CSS. Похожесть — дисциплиной, не общим кодом.
   (Тот же принцип, что для чат-рендереров: общая семантика, раздельная реализация.)
4. **Данные строгие и общие, UI мягкий и локальный — но на класс схемы, а не на файл.**
   У тебя сотни `table_*.jsonl` одной формы; схема (и data, и UI) одна на весь класс. Per-файлового
   override нет (экзотика, в модель не закладываем). Файл данных лишь **декларирует имя** схемы.

## Разрешение схемы — по имени класса, не по файлу/пути

Файл данных декларирует **имя** схемы (стабильный идентификатор с версией), а не путь:

```
data:  sql-table@1     → плагин (авторитет; один на все table_*.jsonl)
UI:    sql-table.ui     → плагин-референс ← опц. проектный override ← автогенерация
```

- Схема **данных** не переопределяется проектом — она про корректность, авторитет у поставщика.
- **UI** опциональна: нет её → редактор разворачивает форму авто (поля по порядку, виджет по типу).
  Плагин кладёт референс «чтоб красиво»; проект может перекрыть его **один раз на класс**.
- Версия в имени (`@1`) — чтобы схема не поехала впереди данных. Заложить сразу, мигрировать дороже.

## Дорожная карта (фазы = слои стека)

- **Фаза 0 — контракты на бумаге (ТЕКУЩАЯ).** Два дока (А, Б) + сигнатуры двух интерфейсов. Без кода.
  Почему первым: всё навешивается на эти контракты; поедут потом — переписывать три слоя.
- **Фаза 1 — Core-редактор** (отдельный проект `SPLA.Editor.Schema`): Avalonia+WebView, React+JSON Forms.
  Тупой, знает два резолвера. **Рендер-решение (зафиксировано):** React+JSON Forms собирается Node'ом
  **только на этапе сборки** в ОДИН self-contained HTML (vite-plugin-singlefile инлайнит весь JS/CSS),
  который вшивается как **EmbeddedResource в DLL**. В готовом приложении: всё внутри DLL, никаких
  `.js/.css/.html/wwwroot` рядом, никакого Node/npm у пользователя. В рантайме HTML достаётся из
  ресурса → temp-файл → `NativeWebView.Navigate` (тот же приём, что `ChatHtmlRenderer.RenderToFile`).
  Контракты (`IJSONSchemaProvider`/`IContentSource`/`JsonSchema`/`SchemaRegistry`) — в `SPLA.Domain`.
- **Фаза 2 — Резолверы**: `SchemaRegistry` (собирает `IJSONSchemaProvider`'ы) + `IContentSource`.
  SQL-плагин регистрирует провайдер с embedded `sql-table` / `sql-table.ui`.
- **Фаза 3 — Workspace-shell** (отдельный проект): дерево + диспетчер редакторов. Детали — §«Фаза 3» ниже.
- **Фаза 4 — SPLA-host + launch-tool**: shell в процессе SPLA, резолверы на проект; инструмент `editor_open`.

Версионирование (`@N`) и порядок разрешения фиксируются в Фазе 2, но заложены уже в Доке А.

## Фаза 3 — Workspace-shell: раскладка и модель открытия

### Раскладка

```
┌──────────────── переключатели сверху ─────────────────┐
│  [Режим: View | Edit]      [jsonl: Forms | CodeMirror] │
├───────────────┬───────────────────────────────────────┤
│ Дерево файлов │  Редактор / просмотр                   │
│ колонки:      │  (по текущим переключателям)           │
│  Имя│Размер│Дата                                       │
└───────────────┴───────────────────────────────────────┘
```

- **Слева** — дерево (`IContentBrowser`), колонки **Имя / Размер / Дата** (ContentNode несёт
  `SizeBytes` + `Modified`).
- **Справа** — редактор/просмотр.
- **Логика клика:** тык на файл → shell по ТЕКУЩИМ переключателям решает, *куда* (Forms/CodeMirror)
  и *как* (View/Edit) грузить, и грузит. Никакого подглядывания в содержимое для авто-выбора.

### Две оси, обе — явные переключатели сверху

| Ось | Значения | Дефолт | Примечание |
|-----|----------|--------|------------|
| **Режим** | View / Edit | **View** | View безопасен (read-only, ничего не пишет). Edit — autosave. |
| **Редактор jsonl** | Forms (`SchemaEditorView`) / CodeMirror (`TextEditorView`) | — | **Явный** выбор пользователя. Авто-логики `$schema→Forms` НЕТ. |

- Любой файл (вкл. `.jsonl`) можно открыть в `TextEditorView` — переключатель это и даёт.
- Прочие текстовые типы (`.md`, `.yaml`, `.json`, `.txt`, неизвестное) → `TextEditorView` (CodeMirror),
  режим подсветки по типу; **preview-пейн** включается только для markdown (сплит внутри web-бандла).

### Сохранение — autosave по потере фокуса

- **Нет** кнопки «Сохранить», нет диалогов «сохранить изменения?».
- В режиме **Edit** запись в `IContentSource.WriteText` происходит **по потере фокуса** редактором
  (ушёл — сохранилось) и при закрытии/смене файла. **НЕ** по таймеру-дебаунсу (чтобы не насиловать SSD).
- **View** не пишет никогда — открыть на просмотр безопасно по определению.

### Расширение файлов SchemaEditorView

Остаётся **`.jsonl`** (стандартный JSON Lines; у нас meta-строка + N строк-полей — это именно JSONL,
не один JSON-документ). Своё расширение НЕ вводим (принцип «соответствовать стандартам»). Какой
редактор открыть для `.jsonl` — решает переключатель Forms/CodeMirror, а не расширение и не содержимое.

### Контракты редакторов (Фаза 3)

`SchemaEditorView` (есть) и `TextEditorView` (CodeMirror + опц. markdown-preview, новый) оба должны
уметь: **View (read-only)** и **Edit + autosave-on-blur**. Shell хранит обе оси на открытый файл,
даёт переключатели и подвешивает запись по blur. Диспетчер редакторов — расширяемый реестр
`ContentType → factory` с текстовым фолбэком (CodeMirror жуёт любой plain text).
