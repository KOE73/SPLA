# DESIGN: SSH live-session tools — поток, экран, диалоги

Статус: принято к реализации (2026-07-14). Контекст: обсуждение после инцидентов
«ubuntu upgrade завис на debconf-диалоге, агент молча ждал» и
«`sudo reboot; echo __SPLA_END__` — сентинел не придёт никогда».

## Проблемы, которые решаем

1. `ssh_session_exec` блокируется до сентинела → интерактивный диалог или reboot вешают turn навсегда; человек вынужден вмешиваться руками.
2. Полноэкранные программы (top, debconf, whiptail, меню со стрелками) для агента нечитаемы: он видит сырой поток escape-кодов, а не экран.
3. Большие выводы (55k+ chars) и повторные «скрины» раздувают контекст.

## Ключевые решения и почему

### 1. Не будить агента — возвращать управление (timeout как у Claude Code Bash)

Инструмент, а не планировщик, отвечает за время. `ssh_session_exec` получает
`timeout_s`; по истечении возвращает НЕ ошибку, а `status: running` + новый вывод.
Turn продолжается, модель решает сама: ждать / отвечать в диалог / спросить человека.
Отвергнуто: «будить агента посреди работы инструмента» (ломает модель turn'а),
субагенты (работа последовательная: жди→смотри→решай, параллелизм не нужен).

### 2. Кольцевой буфер + курсоры читателей

pty пишет весь вывод в ring buffer на сервере (~256KB) непрерывно.
У каждого читателя (агент, живой терминал человека, лог) — свой курсор.
`exec` и `wait` — один механизм чтения с курсора; между вызовами ничего не теряется.

### 3. Сентинел + disconnect-as-result

Конец команды в интерактивном шелле определяется сентинелом
(`; echo __SPLA_END_<rand>__$?`). Разрыв соединения — это тоже РЕЗУЛЬТАТ:
read-loop обязан немедленно вернуть «connection closed by remote (вероятно reboot)»,
а не висеть. Это закрывает кейс reboot: дальше модель в цикле wait/reconnect
с нарастающим интервалом.

### 4. Серверный VT-эмулятор экрана

У SSH-соединения экрана нет — экран существует только у интерпретатора потока.
Рядом с ring buffer тот же поток прогоняется через VT100/xterm-парсер,
поддерживающий сетку Cell[rows,cols] (символ + атрибуты: цвет, инверсия, bold).
Библиотека: начать с готовой (кандидат — VtNetCore, MIT, C#); свой парсер «на 80%»
ломается ровно на тех ncurses-диалогах, ради которых всё делается.

Размер — свойство pty, объявляется при открытии и меняется window-change.
Один авторитетный размер на сессию (показывать в списке сессий, напр. `100×80`).

Resize человеком в произвольный момент — штатно:
- resize идёт через ту же очередь событий сессии, эмулятор атомарно пересоздаёт сетку;
- инструменты НИКОГДА не кешируют координаты/экран между вызовами — каждый шаг
  перечитывает состояние, поэтому resize между шагами безвреден;
- каждый кадр аннотирован своим размером; если размер изменился во время
  send_keys/dialog_choose — шаг просто перечитывает экран и продолжает;
- опционально: пока инструмент в settle-окне (~200мс после нажатия), resize
  откладывается до конца окна.

### 5. Набор инструментов (лесенка от дешёвого к точному)

| Tool | Что делает |
|---|---|
| `ssh_session_exec(cmd, timeout_s)` | пишет команду, ждёт сентинел до таймаута; возвращает новый вывод + status running/done/disconnected |
| `ssh_session_wait(timeout_s, until?)` | читает с курсора дальше до regex/сентинела/таймаута/разрыва |
| `ssh_session_screen()` | отрендеренный экран (24 строки текста) с аннотацией фокуса: выделенные (reverse video) регионы помечены `>>Ok<<`; отдаёт «экран не изменился», если кадр тот же |
| `ssh_session_send_keys(keys)` / `send_text` | стрелки/Tab/Enter/Esc в pty; ВСЕГДА возвращает экран после нажатия (settle ~200мс) + краткий diff («фокус: Ok → Cancel») |
| `ssh_dialog_choose(label)` | намерение вместо механики: детерминированный цикл в C# жмёт Tab/Down, по атрибутам находит выделенный регион, сравнивает с label, Enter; лимит шагов, при зацикливании — ошибка + экран |

Баланс «маленькая сеть vs умный инструмент»: модель говорит «выбери Install»,
перебор фокуса делает код без LLM-токенов. `send_keys` — ручной запасной режим.

### 6. Экономия контекста (без обучения модели — руками хоста)

- Кадры экрана помечаются ephemeral/volatile: при появлении нового кадра той же
  сессии старый заменяется в контексте заглушкой `[кадр экрана 14:32, устарел]`.
  В живом контексте — максимум один кадр на сессию. Использует существующий
  механизм IsEphemeral (ChatMessage → ContextAssembler его отфильтровывает).
- exec/wait возвращают только новый хвост (~2KB) + счётчики; полный лог — в файл
  workspace, в результате ссылка.

### 7. Детектор «завис на вопросе»

После N секунд тишины смотреть не на последнюю строку потока (для ncurses это мусор),
а на отрендеренный экран; отдавать его в ProgressScope («похоже, ждёт ввода: …»)
— человек видит в прогресс-карточке чата, модель получит при таймауте exec.

  машине. Экран и send-keys даёт сам tmux (`capture-pane -p`), сессия переживает
  разрыв соединения (reboot-кейс закрывается идеально). Осознанно отложено:
  требует tmux на remote, добавляет матрицу совместимости. Эмулятор — базовый
  путь без зависимостей; tmux потом как enhancement «если есть — используем».
