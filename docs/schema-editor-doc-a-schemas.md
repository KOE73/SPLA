# Док А — форматы схем и разрешение

Контракт Фазы 0. Описывает три вещи:
1. формат **схемы данных** (что редактируем);
2. формат **UI-схемы** (как разложить, опционально);
3. **разрешение** схемы по имени + сигнатуры резолверов.

Общая карта и принципы — в [плане](schema-editor-plan.md). Токены/стиль — в [Доке Б](schema-editor-doc-b-tokens.md).

---

## 0. Базовые правила

- **ТРЕБОВАНИЕ: схема данных строго соответствует стандарту JSON Schema (draft 2020-12).**
  Документ всегда валиден стандартным валидатором. Самодельные/нестандартные ключи в схему данных
  **запрещены** — иначе теряется соответствие стандарту. См. §1.
- **UI — ОТДЕЛЬНО, всегда.** Раз стандарта на UI-раскладку не существует, UI не примешивается к схеме
  данных и не «пачкает» её нестандартными ключами. Отдельный документ, отдельный канал (`<name>.ui`).
- **Схема резолвится по имени класса**, не по пути файла. Файл данных лишь декларирует имя.
- **Имя версионируется**: `<name>@<major>` (напр. `sql-table@1`). UI-схема того же класса — `<name>.ui`.
- **enum'ы и ограничения — всегда в схеме данных**, никогда в UI. UI решает только подачу.
- Схема данных — авторитет поставщика (плагина). UI — мягкая, опциональная, переопределяемая проектом.
- Ни схема, ни данные не обязаны лежать в ФС — всё ходит через резолверы (см. §4).

## 1. Схема данных (data schema)

**Строгий JSON Schema (draft 2020-12) — соответствие стандарту обязательно.** Документ всегда валиден
стандартным валидатором; нестандартных ключей в нём нет.

Граница между «соответствует стандарту» и «реализуем не всё»:
- **Поддержка рендером — подмножество** ключевых слов (на старте): `type`
  (object/array/string/number/integer/boolean), `properties`, `items`, `required`, `enum`, `title`,
  `description`, `default`, `minimum`/`maximum`, `minLength`/`maxLength`, `pattern`.
- **Игнорировать часть стандартных ключей при рендере — можно** (это не нарушает стандарт: документ
  остаётся валидным, просто рендер не использует всё).
- **Добавлять своё, чего нет в стандарте, — НЕЛЬЗЯ** (это и есть нарушение). Любая UI/визуальная
  информация идёт в UI-схему (§2), а не сюда.

Идентификация: `$id` несёт версионированное имя класса (`"sql-table@1"`).

### Что описывает (важный нюанс)

Редактор правит JSONL-**документ таблицы** (поток строк `{type:meta}` + N×`{type:field}`).
Концептуально это один объект `{ meta, fields[] }`. Схема описывает **форму этого документа**,
а не строку данных таблицы БД. На сохранении редактор раскладывает объект обратно в JSONL-строки.

### Референс `sql-table@1`

```json
{
  "$id": "sql-table@1",
  "title": "Описание таблицы БД",
  "type": "object",
  "properties": {
    "meta": {
      "type": "object",
      "title": "Метаданные таблицы",
      "properties": {
        "table": { "type": "string", "title": "Таблица" },
        "rules": {
          "type": "array", "title": "Правила",
          "items": { "type": "string" }
        },
        "joins": {
          "type": "array", "title": "Связи",
          "items": {
            "type": "object",
            "properties": {
              "to": { "type": "string", "title": "К таблице" },
              "on": { "type": "string", "title": "По полю" }
            },
            "required": ["to", "on"]
          }
        }
      },
      "required": ["table"]
    },
    "fields": {
      "type": "array", "title": "Поля",
      "items": {
        "type": "object",
        "required": ["name", "data_type"],
        "properties": {
          "name":      { "type": "string", "title": "Поле" },
          "data_type": { "type": "string", "title": "Тип" },
          "agg": {
            "type": "string", "title": "Агрегация",
            "enum": ["SUM", "AVG", "MIN", "MAX", "filter", "group", "none"],
            "default": "none"
          },
          "desc": { "type": "string", "title": "Описание" },
          "note": { "type": "string", "title": "Заметка" }
        }
      }
    }
  },
  "required": ["meta", "fields"]
}
```

Профит: `agg` как `enum` → в редакторе выпадашка, кривое значение физически не ввести.
Это и есть «строгость, исключающая расхождения».

### Данные, которым соответствует (тот же JSONL на диске)

```jsonl
{"type":"meta","table":"АсбестКоличество","rules":["не суммировать Тип 0 и Тип 3"],"joins":[{"to":"Продукт_0","on":"Марка"}]}
{"type":"field","name":"OID","data_type":"int","agg":"none","desc":"ID записи"}
{"type":"field","name":"Масса","data_type":"numeric","agg":"SUM","desc":"вес продукта"}
{"type":"field","name":"Тип","data_type":"tinyint","agg":"filter","desc":"тип операции"}
```

Маппинг JSONL ↔ объект схемы: строка `{"type":"meta",…}` → `meta`; каждая `{"type":"field",…}` →
элемент `fields[]` (ключ `type` — дискриминатор строки, в объект схемы не попадает).

## 2. UI-схема (UI schema) — опциональная

Формат — **JSON Forms UI Schema** (layout-дерево: `VerticalLayout`/`HorizontalLayout`/`Group`/`Control`,
ссылки `scope` на `#/properties/...`). По умолчанию **её нет** → редактор генерит авто:
поля по порядку объявления, виджет по типу (`number`→числовое, `boolean`→чекбокс, `enum`→выпадашка).

UI-схема — **частичный патч** поверх авто: описываешь только то, что хочешь изменить (группировку,
порядок, скрытие служебного), остальное остаётся авто.

### Референс `sql-table.ui`

```json
{
  "type": "VerticalLayout",
  "elements": [
    { "type": "Group", "label": "Таблица", "elements": [
      { "type": "Control", "scope": "#/properties/meta/properties/table" },
      { "type": "Control", "scope": "#/properties/meta/properties/rules" }
    ]},
    { "type": "Group", "label": "Связи", "elements": [
      { "type": "Control", "scope": "#/properties/meta/properties/joins" }
    ]},
    { "type": "Control", "scope": "#/properties/fields",
      "options": { "elementLabelProp": "name" } }
  ]
}
```

Тут: meta — в группы сверху, поля — таблицей снизу. Скрыть служебное (`OptimisticLockField`) =
просто не указать его контрол. Данных не трогает — только вид.

**UI-схема — на класс, а не на файл.** Один `sql-table.ui` управляет видом всех `table_*.jsonl`.

## 3. Разрешение (resolution order)

```
data schema:  <name>@<ver>   → поставщик-авторитет (плагин). НЕ переопределяется проектом.
ui schema:    <name>.ui       → плагин-референс  ←  проектный override  ←  автогенерация
content:      <ref>           → через IContentSource
```

- data: первый провайдер, отдавший имя, выигрывает; но по смыслу авторитетом считается поставщик класса.
- ui: сильнее всех проектный override (один на класс), затем плагин-референс, иначе авто.
- Имя + версия декларируются в файле данных (предлагаемо — в meta-строке, поле `$schema`):
  `{"type":"meta","$schema":"sql-table@1", ... }`.

## 4. Сигнатуры резолверов (контракт, код — в Фазе 2)

Два независимых резолвера. Ни один не предполагает файловую систему — это абстракции «достать по имени/
ссылке», физический источник (embedded-ресурс, проект, in-memory от агента) — забота реализации.

```csharp
// Схема по имени класса. Несколько провайдеров (плагин, проект, агент) опрашиваются по очереди.
public interface IJSONSchemaProvider
{
    // Может ли отдать ИМЕННО эту схему (имя включает версию, напр. "sql-table@1";
    // UI-схема — "sql-table.ui").
    bool CanResolve(string schemaName);

    // Вернуть схему как структурированный JSON-документ (data schema или UI schema по тому же имени).
    // Бросать/возвращать null, если !CanResolve.
    JsonSchema Resolve(string schemaName);
}

// Содержимое по ссылке. Не обязательно путь в ФС — это может быть ключ проекта, blob, in-memory.
public interface IContentSource
{
    bool CanResolve(string contentRef);

    // Сырое содержимое редактируемого документа (напр. JSONL-текст таблицы).
    string ReadText(string contentRef);

    // Сохранение правок обратно туда же.
    void WriteText(string contentRef, string text);
}

// Обзор дерева (источник данных для дерева файлов Workspace-shell, Фаза 3). Закрывает разрыв:
// IContentSource читает/пишет ОДНУ ссылку, но не перечисляет. Ref узла = та же опаковая ссылка,
// что потом идёт в IContentSource. ContentType листа — подсказка для выбора редактора (.jsonl/.md).
public interface IContentBrowser
{
    string RootLabel { get; }
    // parentRef == null → корень; иначе Ref ранее выданной ветки. Ленивое раскрытие.
    IReadOnlyList<ContentNode> GetChildren(string? parentRef);
}
// ContentNode(Ref, Label, Kind: Folder|Leaf, ContentType?) — плоский DTO.
```

Реализации поверх ФС: `FileContentSource` (ссылка = путь) и `FileContentBrowser` (выдаёт пути-листья
с ContentType из расширения). Браузер и источник образуют пару: обзор отдаёт Ref → источник читает.

Реестр (Фаза 2) собирает провайдеры и резолвит по первому совпадению:

```csharp
public sealed class SchemaRegistry
{
    public void Register(IJSONSchemaProvider provider);
    public JsonSchema Resolve(string schemaName);   // по порядку регистрации / приоритету
    public bool TryResolve(string schemaName, out JsonSchema schema);
}
```

`JsonSchema` — нейтральная обёртка над JSON-документом. Для **схемы данных** содержимое — всегда
валидный JSON Schema 2020-12 (стандарт). Для **UI-схемы** содержимое — отдельный документ (диалект
выбранного рендера), резолвится по имени `<name>.ui` тем же интерфейсом.

### Решение (бывший «открытый вопрос»)

**Закрыто требованием соответствия стандарту:**
- **Схема данных = строгий JSON Schema 2020-12.** Это стандарт, он рендер-независим по определению.
  Никакого выбора и никакой трансляции данных — берём стандарт как есть. Привязки к библиотеке нет.
- **UI = ОТДЕЛЬНЫЙ документ**, не примешивается к данным. На старте — диалект конкретного рендера
  (напр. JSON Forms uischema), прокидывается в WebView напрямую, без транслятора.
- **Точка трансляции UI заложена, но не строится.** Если когда-нибудь добавится второй рендер —
  транслятор UI вставляется в ОДНОМ месте (в Core-редакторе перед WebView); сигнатуры провайдеров и,
  главное, схемы данных при этом не трогаются.

Итог: данные строго стандартны и неизменны при смене рендера; вся «небратость» изолирована в UI,
который и так отдельный артефакт. Изобретать нейтральный язык раскладки заранее — не нужно.
