# LogSentry — автономный дежурный по логам на SPLA Runtime

Второй демо-хост `SPLA.Runtime` без CLI/сервиса/UI: следит за лог-файлом,
свежие подозрительные строки пачками отправляет агенту, вердикт — в консоль.

## Что делает

Цикл: **хвост лога → фильтр по маркерам → пачка строк → ход агента → вердикт → снова**.

- Промпт триажа — обычные `agent.instructions` проекта ([prompt.md](prompt.md)):
  `ИНЦИДЕНТ | ШУМ | НАБЛЮДАТЬ`, суть и действие.
- В отличие от VisionAgent контекст здесь частично копится (`keep_history: 3`) —
  агент видит предыдущие пачки и замечает нарастание одной и той же ошибки.
- Ротация/усечение лога учитываются: файл перечитывается с начала.

## Запуск

1. LM Studio с любой моделью: `lms server start && lms load <model> --yes`
2. Настройте `sentry.spla` (`log_file`, `triggers`, endpoint, model).
3. `dotnet run --project demo/LogSentry` (или аргументом путь к другому `.spla`).

Остановка — `Ctrl+C`; в конце печатается `готово` / `не готово`.

## Ключи секции `sentry:`

| Ключ | Значение |
|------|----------|
| `log_file` | файл лога (путь относительно `.spla`) |
| `triggers` | маркеры-подстроки; пусто = слать все новые строки |
| `batch_seconds` | сколько секунд копить строки перед отправкой |
| `max_lines` | максимум строк в пачке |
| `keep_history` | сколько прошлых пачек-ходов держать в контексте |
